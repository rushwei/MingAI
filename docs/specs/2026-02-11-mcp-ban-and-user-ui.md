# MCP Key 永久封禁与用户页改版（Full Spec）

## 1. 背景与问题

- 当前现状：
  - 管理员吊销 MCP Key 仅将 `is_active=false`。
  - 用户可通过“重置 MCP Key”重新激活，无法满足永久封禁要求。
  - 用户 MCP 页面当前为卡片式，信息分散，操作路径较长。
- 触发原因：
  - 业务明确要求“封禁后不可通过后续重置恢复”。
  - 业务要求 key 可持续展示，并将页面改为更结构化的“表格感”布局。
- 主要痛点：
  - 封禁策略可被绕过（安全策略失效）。
  - 用户操作（复制/重置）不够集中，关键数据不够直观。

## 2. 目标与非目标

### 2.1 目标

- 实现 MCP Key 永久封禁语义：被管理员吊销后，用户无法再通过重置恢复。
- 保持 key 可持续展示，不再采用“一次性展示”文案/逻辑。
- 将用户 MCP 页面改为表格化信息呈现，并将“重新生成 Key”放在复制按钮旁。

### 2.2 非目标

- 不调整管理员 MCP 管理页整体交互模式。
- 不改动 MCP 服务器地址硬编码策略（按业务要求维持现状）。
- 不引入新的数据库字段或 migration。

## 3. 方案设计

### 3.1 变更范围

- 影响模块：
  - `src/lib/mcp-keys.ts`
  - `src/app/api/user/mcp-key/route.ts`
  - `src/app/user/mcp/page.tsx`
  - `src/tests/mcp-key-logic.test.ts`
  - `src/tests/mcp-ui-completeness.test.ts`
- 影响接口：
  - `PUT /api/user/mcp-key` 在“已被管理员吊销”场景下返回失败，不再恢复 key。
- 影响数据结构：
  - 无数据库结构变更，继续复用 `mcp_api_keys.is_active` 作为封禁状态信号。

### 3.2 实现方案

- 核心思路：
  - 在 `resetMcpKey` 增加“封禁态拦截”：若当前 key `is_active=false`，直接拒绝重置。
  - 用户页移除“仅一次显示”的状态控制，持续展示完整 key。
  - 用户页信息区改为表格化（字段-值）布局；把复制和重置操作放在同一操作区域。
- 关键流程：
  - 管理员吊销 -> `is_active=false`。
  - 用户请求重置 -> 检测到 `is_active=false` -> 返回错误，不执行 update。
  - 用户页面：统一表格展示 key、状态、创建时间、上次使用时间，并提供并排操作按钮。
- 兼容策略（向前/向后兼容）：
  - 旧 key 数据无需迁移。
  - 已被吊销用户将从“可重置恢复”变为“永久封禁”，属于预期策略收紧。

### 3.3 数据库与迁移（如适用）

- 是否需要 migration：否。
- 涉及对象（表/字段/索引/约束）：继续使用 `mcp_api_keys.is_active`。
- 回填策略：不需要。
- RLS/权限影响：无新增；仍通过既有服务端受控访问执行。

## 4. 验收标准

- 功能验收：
  - 管理员吊销后，用户调用重置接口失败。
  - 用户页面 key 持续可见。
  - 用户页面以表格化结构呈现关键数据，复制与重置按钮相邻。
- 接口验收：
  - `PUT /api/user/mcp-key` 在封禁场景返回明确错误信息。
- 性能/稳定性验收：
  - 不新增外部依赖与重型计算。
- 回归范围：
  - MCP Key 生成/查询/重置/吊销逻辑。
  - 用户 MCP 页面主要交互与文案。

## 5. 测试计划

- 单元测试：
  - 新增 `resetMcpKey` 在 `is_active=false` 场景下拒绝重置的测试。
- 集成/路由测试：
  - 复用现有 API 逻辑测试，验证错误返回路径。
- 手动验证步骤：
  - 管理员吊销某用户 key。
  - 该用户访问页面确认 key 状态为吊销。
  - 点击重置应失败且提示封禁。
  - 检查页面表格化展示和复制/重置并排布局。

## 6. 风险与回滚

- 失败信号：
  - 吊销后仍可成功重置。
  - 用户页按钮位置或表格布局异常。
- 风险等级：中。
- 回滚步骤：
  - 回滚本次变更提交，恢复旧重置逻辑与旧页面布局。
