# 业务命理逻辑一致性修复与优化

## 1. 背景与问题

- 当前现状：Web 与 MCP 端存在多处算法口径不一致（六爻起卦、八字/紫微真太阳时、奇门盘式参数、塔罗牌库/牌阵等），且部分路由缺少输入校验与时区策略。
- 触发原因：双实现长期演进、参数未贯通、随机种子作用域设置错误、输入校验缺失。
- 主要痛点：跨端结果不可比、回归风险高、边界条件易错、用户体验不一致。

## 2. 目标与非目标

### 2.1 目标

- 统一算法权威口径（以 mcp-core 为准），Web 端尽量薄封装。
- 引入显式时区与可选真太阳时策略，减少时区导致的盘面偏差。
- 补齐输入校验与路由边界，覆盖关键回归测试。

### 2.2 非目标

- 不新增数据库主表或重构业务目录结构。
- 不改变 AI 解读话术或输出文本风格（除必要字段同步）。

## 3. 方案设计

### 3.1 变更范围

- 影响模块：六爻、八字、紫微、奇门、大六壬、塔罗、合盘、MBTI、面相、手相。
- 影响接口：相关 API 路由与 MCP 工具 schema（新增可选字段）。
- 影响数据结构：新增 `timezone` / `longitude` 入参；塔罗与六爻结果结构做小幅字段补充。

### 3.2 实现方案

- 核心思路：
  - 六爻、奇门、大六壬、塔罗等以 mcp-core 为权威输出；Web 端仅做输入校验与展示映射。
  - 时间输入统一为显式时区（默认 `Asia/Shanghai`），经度存在时启用真太阳时。
- 关键流程：
  - 六爻三钱起卦按 6/7/8/9 判定动爻与阴阳。
  - 奇门 `panType` 贯通至核心排盘；大六壬按时区构造 Date。
  - 合盘使用严格八字排盘替代简化公式。
- 兼容策略（向前/向后兼容）：
  - `timezone`/`longitude` 为可选字段；未提供按默认行为执行。

### 3.3 数据库与迁移（如适用）

- 是否需要 migration：否。
- 涉及对象（表/字段/索引/约束）：无结构变更。
- 回填策略：不适用。
- RLS/权限影响：无。

## 4. 验收标准

- 功能验收：同一输入在 Web/MCP 结果一致（六爻、奇门、塔罗），时区差异不再导致盘面偏移。
- 接口验收：新增输入字段可用且不破坏旧请求；非法输入返回 400。
- 性能/稳定性验收：无显著性能回退。
- 回归范围：六爻/八字/紫微/奇门/大六壬/塔罗/合盘相关页面与路由。

## 5. 测试计划

- 单元测试：六爻三钱映射、合盘八字一致性、时区转换工具、随机种子作用域。
- 集成/路由测试：MBTI/面相/手相输入校验、奇门参数贯通。
- 手动验证步骤：各模块用例各 1 条进行对比验证。

## 6. 风险与回滚

- 失败信号：跨端结果差异未收敛、时区导致日柱错位、抽牌结果异常。
- 风险等级：中。
- 回滚步骤：恢复原起卦逻辑与参数处理，回滚到前一版本。

## 7. 里程碑与任务拆分

- M1：补齐测试与输入校验。
- M2：统一六爻/奇门/大六壬时间策略与参数贯通。
- M3：塔罗与合盘口径对齐并完成回归。
