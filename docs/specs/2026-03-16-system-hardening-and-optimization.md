# 系统稳定性与可维护性优化（Full Spec）

## 1. 背景与问题

- 当前现状：核心聊天流式链路、知识库检索与服务端特权客户端存在稳定性与一致性隐患。
- 触发原因：审查发现流式异常未取消上游、知识库权重缓存无界增长、错误日志吞没、特权客户端行为不一致。
- 主要痛点：成本浪费、内存增长、排障困难、行为不一致导致回归风险。

## 2. 目标与非目标

### 2.1 目标

- 流式失败时确保取消上游流，避免无效 token 消耗。
- 知识库权重缓存加上限与过期清理，避免长期进程内存增长。
- 关键 catch 添加可观测日志，保证排障线索。
- 统一特权客户端在缺失系统管理员凭据时的行为边界（生产 fail-fast，非生产可降级）。
- 对聊天提示词上下文构建进行小规模重构，提升可读性且不改行为。

### 2.2 非目标

- 不改动 API 路由入参、响应结构或数据库结构。
- 不引入新的外部依赖。

## 3. 方案设计

### 3.1 变更范围

- 影响模块：
  - `src/lib/server/chat/stream-response.ts`
  - `src/lib/knowledge-base/search.ts`
  - `src/lib/supabase-server.ts`
  - `packages/mcp-server/src/supabase.ts`
  - `src/lib/server/chat/prompt-context.ts`
  - 相关测试文件
- 影响接口：无
- 影响数据结构：无

### 3.2 实现方案

- 核心思路：在关键失败路径上补齐资源释放与日志；对内存缓存加限；统一缺失凭据行为；对复杂函数拆解为可读小函数。
- 关键流程：
  - 流式异常时调用 `reader.cancel()`，确保上游停止产出。
  - KB 权重缓存按 TTL 与最大容量清理，超限时淘汰最旧条目。
  - KB rerank 与额外候选异常时输出警告日志。
  - 生产环境缺失系统管理员凭据直接抛错，非生产环境仅告警并降级。
- 兼容策略（向前/向后兼容）：保持原有 API 行为与响应结构不变。

### 3.3 数据库与迁移（如适用）

- 是否需要 migration：否
- 涉及对象（表/字段/索引/约束）：无
- 回填策略：无
- RLS/权限影响：不改变策略，仅统一缺失凭据行为

## 4. 验收标准

- 功能验收：流式异常后上游被取消；KB 缓存不再无界增长。
- 接口验收：现有 API 响应结构保持兼容。
- 性能/稳定性验收：无新增明显延迟；出现异常时有日志可追踪。
- 回归范围：聊天流式、KB 检索、特权客户端相关路径。

## 5. 测试计划

- 单元测试：
  - 流式异常扣费失败时取消上游。
  - KB 权重缓存含清理逻辑与日志输出（结构性测试）。
- 集成/路由测试：沿用既有聊天与 KB 相关测试。
- 手动验证步骤：
  - 触发流式异常，确认前端收到错误且上游停止。
  - KB 检索在异常条件下输出警告日志。

## 6. 风险与回滚

- 失败信号：流式请求挂起、KB 检索异常无日志、服务端启动即报错。
- 风险等级：中
- 回滚步骤：回退本次改动文件并恢复旧行为。

## 7. 里程碑与任务拆分

- M1：补齐测试与流式取消逻辑
- M2：KB 缓存与日志优化
- M3：统一特权客户端行为与提示词上下文小型重构
