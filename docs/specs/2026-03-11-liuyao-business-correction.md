# Full Spec: 六爻业务纠偏与去评分统一

## 1. 背景与问题

- 当前 Web 与 MCP 存在两套六爻规则实现，纳甲、换日、伏神、应期等口径不一致。
- Web 端纳甲按单爻阴阳切换，MCP 端按上下卦固定纳甲，导致同一卦在两个入口会得到不同六亲和用神。
- Web 端采用 `子初换日`，MCP 端未统一该规则，导致 23:00 后起卦可出现不同日柱、旬空、六神。
- 当前用神与应期链路依赖 `score/strengthScore/rankScore/confidence` 等数值字段，业务上过度量化。
- 无明确问题时，系统仍允许正式断卦并输出吉凶与应期，不符合“一事一占”。
- 婚恋类目标映射被简化为 `妻财=感情婚姻`，未体现传统“男测财、女测官”与角色差异。

## 2. 目标与非目标

### 2.1 目标

- 统一 Web 与 MCP 六爻核心规则，实现单一共享引擎。
- 以经典六爻常用规则为准修正：纳甲、换日、动变、伏神、三合/六冲、用神选择。
- 移除六爻全部评分和置信度字段，改为定性结构化证据链。
- 无问题时仅允许起卦保存，不允许正式分析与 AI 解卦。
- 修正文案、提示词、MCP schema、UI 选择器与测试，使对外口径一致。

### 2.2 非目标

- 不新增数据库主表或核心目录。
- 不扩展新的六爻流派切换能力。
- 不在本次新增更复杂的用户画像持久化字段；婚恋取用仅做流程辅助与最终目标选择。
- 不扩大改造到八字、塔罗等其他功能的评分系统。

## 3. 规则基线

### 3.1 总体规则

- 规则源以《增删卜易》《卜筮正宗》常见口径为准。
- 默认换日规则采用 `子初换日`。
- 起卦分析必须满足：问题明确、分析目标明确；否则不得输出正式断卦结果。

### 3.2 纳甲与六亲

- 纳甲统一按上下卦固定纳甲映射。
- 不允许按“单爻阴阳切换 `naJiaYang/naJiaYin`”计算六爻纳甲。
- 六亲继续以“宫为我、爻为彼”的五行生克关系计算。

### 3.3 动变与结构规则

- `化进/化退` 使用固定配对规则，不使用地支序号差值通算。
- `三合` 必须满足明确成局条件后方可作为强证据，不得以固定加分直接覆盖旺衰。
- `六冲`、`空亡`、`动空不空`、`冲空不空`、`临建不空` 均为条件性证据，不得写成无条件吉凶结论。
- `伏神` 与 `明现爻` 必须严格区分；伏神只允许输出“伏于某爻下”，不得伪装成该爻本身的主用神。

### 3.4 用神与婚恋场景

- 用神选择改为规则链定性比较，不再用数值排序。
- 同类并见时，比较顺序为：明现/伏藏、月日生克、空亡真假、动而吉凶、化进退/回头生克、与场景相关的世应关系。
- 若规则链无法唯一确定主用神，输出“并看/待辨”，而不是伪造唯一主用神。
- 用神取用顺序补充为：
  - 本卦明现用神优先；
  - 本卦无用神而动爻变出用神时，优先取变爻所化之用神，不先落伏神；
  - 本卦与变爻俱无用神时，若月建/日辰可代用，则先以月建/日辰为用；
  - 飞爻、变爻、月建、日辰俱不能稳定取用时，才转取伏神。
- 变出用神与月建/日辰代用都必须在结构化输出中显式标记来源，不能伪装成“明现于某爻”的普通主用神。
- 婚恋问题不得再将 `妻财` 直接描述为“感情婚姻”唯一目标。
- Web 与 MCP 文案统一改为：
  - `妻财`：钱财、资源、交易、男测对象等；
  - `官鬼`：事业、规则、压力、病灾、女测对象等；
  - 婚恋歧义场景允许同时传 `妻财 + 官鬼`。

### 3.5 旬空、伏神与三合补充规则

- 空亡状态不是单一互斥结论：
  - 动空、冲空、临建可同时构成“由空转实”的条件链；
  - 结构化输出至少要保留“当前主状态 + 触发条件”的信息，不能只剩一个 `kong_changing` 覆盖全部。
- 应期建议必须按当前状态分层：
  - 伏神未出、静空未填、月破/日破未解时，不先输出泛化的“逢某日/月即应”。
  - 先输出解除条件，再输出可能的应期触发项。
- 三合分析不能只保留首个命中的组合。
  - 若同一盘同时存在多个三合迹象或多个可成局组合，输出层必须完整保留，不得按代码顺序静默丢弃。

## 4. 方案设计

### 4.1 共享引擎

- 在 `packages/mcp-core/src` 新增共享六爻核心模块。
- Web 端 `src/lib/divination/liuyao.ts` 改为薄包装：保留起卦交互辅助与页面类型导出，核心分析逻辑委托共享模块。
- MCP `handlers/liuyao.ts` 改为调用同一共享模块，禁止再维护平行规则。

### 4.2 输出结构调整

- 移除所有六爻评分/置信度字段：
  - `YaoStrength.score`
  - `YongShenCandidate.strengthScore`
  - `rankScore`
  - `TimeRecommendation.confidence`
- 保留定性结构化字段：
  - 旺衰等级；
  - 空亡状态；
  - 动静状态；
  - 证据数组；
  - 用神主选/候选/并看状态；
  - 定性应期建议。
- 用神候选 `source` 扩展为：
  - `visible`：本卦明现；
  - `changed`：动爻所化；
  - `temporal`：月建/日辰代用；
  - `fushen`：伏神。
- `selectionStatus` 扩展为：
  - `resolved`
  - `ambiguous`
  - `from_changed`
  - `from_temporal`
  - `from_fushen`
  - `missing`
- `timeRecommendations` 由“日期窗口+置信度”改为“触发条件+证据+说明”：
  - 示例：`逢子日/月`、`待出伏`、`待冲空`、`逢合而动`。
- `sanHeAnalysis` 在兼容原 `fullSanHe` 的同时，新增列表型字段保留全部命中的三合迹象/成局结果。

### 4.3 Web 流程

- `question` 为空时：
  - 允许 `save`
  - 禁止 `interpret`
  - 结果页只展示卦象、起卦时间、卦辞等基础信息
  - 不展示用神、神系、风险、应期、AI 深度解卦入口
- 婚恋类目标选择增加辅助引导，但最终仍只落 `yongShenTargets`。
- 历史旧记录在补录 `question + yongShenTargets` 前，不允许正式分析。

### 4.4 MCP 与文档

- MCP `liuyao_analyze` 输入描述更新为经典优先口径。
- 输出 schema、Markdown formatter、manual、workflow 文档同步删除评分/置信度与错误映射。
- 聊天人格和 API prompt 改为条件性规则表达，不得把三合、六冲、空亡写成绝对判词。

## 5. 验收标准

- 同一卦、同一时间、同一目标，Web 与 MCP 分析结果一致。
- `2024-01-02 23:30 +08:00` 等边界时刻，Web 与 MCP 日柱一致。
- `天风姤`、`巽为风`、`坤为地` 等真值用例的纳甲与六亲符合既定规则表。
- 公共输出、类型、schema、页面文案、Markdown 中不再出现六爻评分和置信度。
- 无问题记录不能正式断卦。
- 婚恋目标映射不再将 `妻财` 直接等价为所有感情婚姻问题。

## 6. 测试计划

- 单元测试：
  - 纳甲真值测试
  - 换日边界测试
  - 动变类型测试（化进/化退/回头生/回头克/化墓/化绝/伏吟/反吟）
  - 用神“并看”与伏神可用性测试
  - 变出用神优先于伏神测试
  - 日月代用优先于伏神测试
  - 空亡层次与应期去冲突测试
  - 多三合并存输出测试
- 一致性测试：
  - Web 与 MCP 同盘输出一致测试
- 路由/UI 测试：
  - 空问题记录可保存不可分析
  - 历史旧记录需补录问题和目标后才可分析
  - 婚恋目标文案与提示词回归测试
- 文本输出测试：
  - 不暴露 `score/rank/confidence/参考度`

## 7. 风险与回滚

- 风险：
  - 共享引擎重构会影响 Web 与 MCP 两侧输出形状。
  - 旧测试大面积依赖评分字段与时间窗口，需要同步调整。
- 回滚：
  - 以本次提交为边界整体回滚，不做半回滚。
