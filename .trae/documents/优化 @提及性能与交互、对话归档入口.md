## 目标与现状判断
- 你现在看到“可以获取但很慢”，根因是 @提及弹层每次打开都会拉 `/api/data-sources`，而这个接口会串行扫一遍所有 provider（八字/紫微/塔罗/六爻/…）做多次数据库查询；数据量越大延迟越明显。
- 你的 4 条需求可以拆成：性能（缓存/增量）、对话列表动作面板（归档入口）、移动端提及按钮、提及弹层层级导航。

## 1）@数据获取慢：做“数据索引缓存 + 并行 + 增量失效”
- **服务端并行化**：把 `getUserDataSourcesWithErrors` 从串行 for-loop 改为 `Promise.all` 并行执行各 provider.list；每个 provider 的错误独立收集，不拖慢整体。
  - 涉及文件：`src/lib/data-sources/index.ts`
- **限制返回量（默认）+ 可配置**：对 @提及来说通常不需要“全量”，而是“最近 N 条 + 计数”。
  - 默认：每个 provider.list 增加 `.limit(50)`（或 ctx.limit），减少大表扫描。
  - 可选增强：接口支持 `?limit=50&includeCounts=1`，每类额外走一次 `select('*', { count: 'exact', head: true })` 只拿 count，不拉全量。
  - 涉及文件：各 provider（`src/lib/data-sources/*.ts`）+ `/api/data-sources` route
- **本地缓存（你说的“维护一个本地全部数据”）的现实落地方式**：
  - 第一阶段先缓存“索引/摘要”（DataSourceSummary 列表 + KB 列表），让 @ 提及与历史列表秒开；不把所有正文/对话消息全量塞 localStorage（体积/隐私/性能风险大）。
  - 具体实现：新建 `useCachedUserDataIndex(userId)` hook：
    - 首次：读 localStorage（带 TTL，例如 5–10 分钟）立即渲染；
    - 后台：刷新 `/api/data-sources?limit=50` 并写回缓存；
    - MentionPopover 改成“只渲染，不再自己 fetch”，避免每次打开都请求。
  - 涉及文件：`src/components/chat/ChatComposer.tsx`、`src/components/chat/MentionPopover.tsx`（以及新增 hook 文件）
- **增量更新/删除触发本地更新**：
  - 统一事件：成功新增/删除六爻/塔罗/合盘等后，`window.dispatchEvent(new CustomEvent('mingai:data-index:invalidate', { detail: { types: [...] } }))`。
  - hook 监听该事件并刷新；跨 tab 可选用 `storage` 事件或 `BroadcastChannel`。
  - 涉及文件：各“保存/删除”的页面（history/result 页）与新增 hook

## 2）对话归档入口：把“重命名/删除”改为截图式动作面板（含归档）
- **UI 交互**：每条对话右侧改成“…”按钮（移动端不依赖 hover），点击弹出底部 Action Sheet：
  - 动作：分享（可选）、重命名、归档（加入知识库）、删除（危险项红色 + 二次确认）。
- **归档实现**：复用现有 `AddToKnowledgeBaseModal`，传 `sourceType='conversation'`、`sourceId=conv.id`、`sourceTitle=conv.title`。
- **展示状态**：如果 `Conversation` 已带 `isArchived/archivedKbIds`，在列表项或 sheet 中显示“已归档”。
- 涉及文件：`src/components/chat/ConversationSidebar.tsx`（主改）+ 复用 `src/components/knowledge-base/AddToKnowledgeBaseModal.tsx`

## 验证方式
- 性能：打开提及弹层时首屏先用本地缓存秒出列表，后台刷新；数据源接口耗时显著下降（并行+limit）。
- 交互：移动端“…”动作面板可重命名/归档/删除；输入框“提及”按钮可打开 @；Esc/返回按钮能逐级回退。

- 本地维护的是“索引/摘要（列表）+ 必要字段”，不是把所有对话消息全文离线缓存。
